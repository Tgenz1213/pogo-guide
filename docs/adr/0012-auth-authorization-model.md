## Context

The platform has two separately deployed surfaces with their own notion of "admin" — the public web app and the Sanity Studio CMS — that are easy to conflate because they share the word "admin" but not an implementation.

`INITIAL_ADMIN_EMAILS` originally only ever granted admin status; removing an email from it never demoted the corresponding user, so it behaved as an append-only bootstrap grant rather than a live allowlist (issue #52). Fixing that raised a second problem: an unconditional "not on the allowlist → demote" rule would also strip admins promoted through the in-app admin panel, since panel grants have no allowlist entry at all. A third requirement followed from that: the repo owner must never be lockable out of their own app by a contributor with admin-panel access, whether via demotion, ban, or warn.

## Decision

Authentication on the web app is via `nuxt-auth-utils` with Discord and Google OAuth providers (`web/server/routes/auth/`). Web-app admin status (`isEmailAdmin`, checked against `INITIAL_ADMIN_EMAILS`) and Studio admin status (`isAdministrator`, checked against Sanity user roles) are two independent authorization systems and must never be treated as interchangeable in code, comments, or docs. Every OAuth provider callback route must determine and self-heal a user's admin status using the single shared `isEmailAdmin()` utility (`web/server/utils/admin.ts`) — no provider callback may reimplement allowlist parsing inline, and no provider may skip re-checking admin status on a returning user's login. Post-login redirect targets must always be passed through `sanitizeRedirectPath` (`web/shared/utils/auth.ts`) before use; its layered checks (raw pattern check, decode-then-recheck, `parseURL` host/protocol check) are deliberate defense-in-depth against open-redirect and protocol-relative bypasses and must not be simplified without understanding why each layer exists.

**`INITIAL_ADMIN_EMAILS` is a live allowlist, not a one-time bootstrap grant** — but only for admins whose current grant came from that allowlist. The `users.adminGrantedVia` column (`"bootstrap" | "admin_panel" | "super_admin" | null`) records _how_ a user's current admin status was granted, so the login self-heal path (`reconcileBootstrapAdmin()`) can distinguish "grant via allowlist, now demotable" from "grant via the admin panel, must never be auto-demoted" from "legacy row predating this column (`null`), treated as unknown provenance and never auto-demoted." No backfill runs on the `admin_granted_via` migration — every admin that existed before this feature is grandfathered in as un-demotable by design.

**`SUPER_ADMIN_IDS`** (a comma-separated list of exact `${provider}:${providerAccountId}` D1 ids, parsed the same way as `INITIAL_ADMIN_EMAILS`) names accounts that are always admin regardless of the allowlist, and can never be demoted, warned, or banned by anyone — including another super admin (mutual immunity). This is enforced by `isSuperAdminId()`/`enforceSuperAdmin()` in `web/server/utils/admin.ts` on login, and by `action.post.ts` rejecting `warn`/`ban`/`make_admin`-revoke against any `SUPER_ADMIN_IDS` entry. **`SUPER_ADMIN_IDS` must be set in every deployed environment** (`wrangler secret put SUPER_ADMIN_IDS`) with at least the repo owner's account id — with it unset, `isSuperAdminId` always returns `false`, and per the next paragraph's hierarchy rule, that means _no admin can moderate any other admin at all_ until it's configured. This fail-closed default is deliberate (favors "moderation temporarily disabled" over "any admin can demote any other admin"), but it is a silent behavior change on first deploy if left unset.

**Admin hierarchy in `action.post.ts`:** a regular (non-super) admin cannot `warn`, `ban`, or revoke admin status from any other admin, super or regular — only a super admin can act against another admin. Any admin retains unrestricted power over non-admin users. The two checks run in a fixed order: a super-admin target is rejected unconditionally first (regardless of who's acting), then a regular-admin target is rejected unless the acting admin is itself a super admin.

## Consequences

- **Pros:** Clear separation between the two admin surfaces prevents accidental privilege bleed between the web app and Studio; centralizing allowlist logic in one utility prevents provider-specific drift. Provenance tracking lets the allowlist actually revoke access without an unconditional demotion rule silently stripping panel-granted admins. `SUPER_ADMIN_IDS` gives the repo owner a lockout-proof account independent of any allowlist or in-app action.
- **Cons:** An admin who needs access to both surfaces must be granted it separately in each system. `isEmailAdmin()` and `sanitizeRedirectPath` become single points that every OAuth provider integration must correctly call — adding a new provider without wiring both is a defect. `SUPER_ADMIN_IDS` being unset disables all admin-on-admin moderation, not just protection of a specific account — this must be configured before or immediately after this feature deploys. Legacy admins (`adminGrantedVia: null`) can never be auto-demoted by the allowlist path even if removed from `INITIAL_ADMIN_EMAILS`, until they're re-granted through a tracked path (admin panel or a fresh bootstrap grant).
