## Context

The platform has two separately deployed surfaces with their own notion of "admin" — the public web app and the Sanity Studio CMS — that are easy to conflate because they share the word "admin" but not an implementation.

`INITIAL_ADMIN_EMAILS` originally only ever granted admin status; removing an email from it never demoted the corresponding user, so it behaved as an append-only bootstrap grant rather than a live allowlist (issue #52). Fixing that raised a second problem: an unconditional "not on the allowlist → demote" rule would also strip admins promoted through the in-app admin panel, since panel grants have no allowlist entry at all. A third requirement followed from that: the repo owner must never be lockable out of their own app by a contributor with admin-panel access, whether via demotion, ban, or warn.

## Decision

Authentication on the web app is via `nuxt-auth-utils` with Discord and Google OAuth providers (`web/server/routes/auth/`). Web-app admin status (`isEmailAdmin`, checked against `INITIAL_ADMIN_EMAILS`) and Studio admin status (`isAdministrator`, checked against Sanity user roles) are two independent authorization systems and must never be treated as interchangeable in code, comments, or docs. Every OAuth provider callback route must determine and self-heal a user's admin status using the single shared `isEmailAdmin()` utility (`web/server/utils/admin.ts`) — no provider callback may reimplement allowlist parsing inline, and no provider may skip re-checking admin status on a returning user's login. Post-login redirect targets must always be passed through `sanitizeRedirectPath` (`web/shared/utils/auth.ts`) before use; its layered checks (raw pattern check, decode-then-recheck, `parseURL` host/protocol check) are deliberate defense-in-depth against open-redirect and protocol-relative bypasses and must not be simplified without understanding why each layer exists.

**`INITIAL_ADMIN_EMAILS` is a live allowlist, not a one-time bootstrap grant** — but only for admins whose current grant came from that allowlist. The `users.adminGrantedVia` column (`"bootstrap" | "admin_panel" | "super_admin" | "revoked" | null`) records _how_ a user's current admin status was granted, so the login self-heal path (`reconcileBootstrapAdmin()`) can distinguish "grant via allowlist, now demotable" from "grant via the admin panel, must never be auto-demoted" from "legacy row predating this column (`null`), treated as unknown provenance and never auto-demoted." No backfill runs on the `admin_granted_via` migration — every admin that existed before this feature is grandfathered in as un-demotable by design.

**A panel revoke is sticky.** `action.post.ts`'s `make_admin` revoke sets `adminGrantedVia: "revoked"` (not `null`). `reconcileBootstrapAdmin()`'s grant branch treats `"revoked"` the same as `"admin_panel"`/`"super_admin"` — it will never silently re-grant that account via the allowlist again, even if their email is still on (or is re-added to) `INITIAL_ADMIN_EMAILS`. Only an explicit admin-panel re-grant, or a `SUPER_ADMIN_IDS` designation, restores admin. Before this, a panel revoke could be silently undone on the very next login if the target's email was still allowlisted — `null` meant both "never granted" (should auto-grant) and "just revoked" (should not), which were indistinguishable.

**`SUPER_ADMIN_IDS`** (a comma-separated list of exact `${provider}:${providerAccountId}` D1 ids, parsed the same way as `INITIAL_ADMIN_EMAILS`) names accounts that are always admin regardless of the allowlist, and can never be demoted, warned, or banned by any _other_ account — including another super admin (mutual immunity). This is enforced by `isSuperAdminId()`/`enforceSuperAdmin()` in `web/server/utils/admin.ts` on login (both orchestrated by the shared `resolveReturningUserAdmin()`, used by both OAuth callbacks so the precedence can't drift between providers), and by the shared `isProtectedFromActor()` primitive that every admin-panel mutation route must call before acting on another account (currently `action.post.ts`'s `warn`/`ban`/`make_admin`-revoke, and `deletion-requests/action.post.ts`'s account-deletion approval). **`SUPER_ADMIN_IDS` must be set in every deployed environment** (`wrangler secret put SUPER_ADMIN_IDS`) with at least the repo owner's account id — with it unset, `isSuperAdminId` always returns `false`, and per the next paragraph's hierarchy rule, that means _no admin can moderate any other admin at all_ until it's configured. This fail-closed default is deliberate (favors "moderation temporarily disabled" over "any admin can demote any other admin"), but it is a silent behavior change on first deploy if left unset.

`SUPER_ADMIN_IDS` (an env var) and `adminGrantedVia = "super_admin"` (a D1 column, only ever written by `enforceSuperAdmin()` at login) are two independent representations of "who is a super admin" that can disagree between logins — editing the env var doesn't retroactively update rows for accounts that haven't logged in since. There is no single query that answers "list every super admin" without cross-referencing both; the env var is the actual source of truth, the DB column is a lazily-updated cache of the most recent login's answer.

**Admin hierarchy:** a regular (non-super) admin cannot `warn`, `ban`, revoke admin status from, or delete the account of any _other_ admin, super or regular — only a super admin can act against another admin. Any admin retains unrestricted power over non-admin users. `isProtectedFromActor()` has **no self-action exemption** — it runs the same two checks regardless of whether the actor and target are the same account: a super-admin target is rejected unconditionally, then a regular-admin target is rejected unless the acting admin is itself a super admin. This is deliberate: a self-`ban` would insert a row into `banned_identities`, which the OAuth login flow checks _before_ `enforceSuperAdmin` runs, so it would permanently lock the account out with no self-heal — unlike a self-demotion (`isAdmin` flips back on next login via `enforceSuperAdmin`) or a self-approved GDPR deletion (the row is simply recreated with super-admin status on next login), neither of which touches `banned_identities`. Callers that want to allow a _specific_ self-directed action exempt `actingUserId === target.id` themselves, scoped narrowly to that action: `action.post.ts`'s `make_admin` revoke skips `assertNotProtected()` entirely when the target is the acting admin (self-demotion), but `warn` and `ban` never do; `deletion-requests/action.post.ts`'s approve path skips it only when the request being approved is the actor's own. Every route that lets one admin mutate another account must still call `assertNotProtected()` (the shared wrapper around `isProtectedFromActor()`, which throws the standard 403) for every action that isn't one of these narrow, deliberately-chosen exemptions — a route that skips it, or over-broadens a self-exemption, silently reopens the exact lockout risk this feature exists to close.

## Consequences

- **Pros:** Clear separation between the two admin surfaces prevents accidental privilege bleed between the web app and Studio; centralizing allowlist logic in one utility prevents provider-specific drift. Provenance tracking lets the allowlist actually revoke access without an unconditional demotion rule silently stripping panel-granted admins, and a sticky "revoked" state means a panel revoke can't be silently undone by the allowlist either. `SUPER_ADMIN_IDS` gives the repo owner a lockout-proof account independent of any allowlist or in-app action, and `isProtectedFromActor()` being a shared primitive means every current admin-mutation route (including account deletion, not just warn/ban/revoke) enforces the same hierarchy.
- **Cons:** An admin who needs access to both surfaces must be granted it separately in each system. `isEmailAdmin()` and `sanitizeRedirectPath` become single points that every OAuth provider integration must correctly call — adding a new provider without wiring both is a defect. `SUPER_ADMIN_IDS` being unset disables all admin-on-admin moderation, not just protection of a specific account — this must be configured before or immediately after this feature deploys. Legacy admins (`adminGrantedVia: null`) can never be auto-demoted by the allowlist path even if removed from `INITIAL_ADMIN_EMAILS`, until they're re-granted through a tracked path (admin panel or a fresh bootstrap grant). `SUPER_ADMIN_IDS` and the DB's `super_admin` tag can disagree between logins (see above) — there's no live query for "current super admins," only "who was a super admin as of their last login."
